Ingestion & feature engineering

The IBM Telco Churn dataset (7,043 customers, 26.5% churn rate) loaded via Delta Lake with schema and null-rate validation logged to MLflow for lineage. Five engineered features sit on top of the raw columns — tenure buckets, ordinal contract encoding, an add-on engagement count, and a charge-increase ratio that flags recent price hikes, a known churn trigger.

why one dataset, not two A second real telecom dataset was evaluated for drift detection, but only ~7 features mapped cleanly across both schemas — dropping the strongest predictors and cutting expected AUC from ~0.84 to ~0.70. Drift detection instead uses synthetic injection (see section 06), the same pattern used by Evidently AI and Databricks' own MLOps references.

Model registry & validation gate

The best run is found by querying the MLflow experiment directly — not reused from an in-memory variable — mirroring how a real promotion stage runs independently of training. Before promotion, the model is scored on the held-out test set for the first and only time, gated at test_roc_auc ≥ 0.80.

07

Batch scoring

Reads the champion pointer fresh on every run — no code change needed after a retrain promotes a new champion. Scores customers, assigns risk tiers, and appends to Delta with full lineage (model_run_id, scored_at) for auditability. This notebook is the scheduled job task, configured for a daily 6am run and currently paused.

design

Key decisions

sklearn over Spark ML

7K rows fits comfortably in pandas; SHAP's mature tooling targets sklearn directly. Spark handles ingestion and feature engineering at scale instead.

manual MLflow logging

Autolog is demoed for fast exploration, but the systematic comparison uses manual logging for consistent metric names and custom artifacts across model families.

synthetic drift injection

Preserves the full, high-signal feature set for modeling while still exercising a genuine, working drift-detection and retrain pipeline.
